# BiliFocus 开发流程

## 1. 总原则

这个项目不是靠“边写边猜”推进的。

推荐顺序是：

1. 先明确范围
2. 再冻结契约和数据模型
3. 再改后端与前端实现
4. 最后补 README、验收和运行说明

不要跳过契约和模型，直接开始写页面或接口。

## 2. 变更顺序

当变更涉及共享结构时，统一按下面顺序处理：

1. 更新相关文档
2. 更新后端 schema / service / repository
3. 更新前端类型与 API 客户端
4. 更新页面、组件和交互
5. 更新测试、验收说明或 README

适用范围：

- 新接口
- 响应字段变更
- 持久化字段变更
- 用户可见能力变更

## 3. 推荐实施顺序

### 阶段一：范围与契约

- 明确本次需求是否超出当前项目范围
- 明确是否影响 `docs/api-contract.md`
- 明确是否影响 `docs/data-model.md`

### 阶段二：后端能力

- 先补 service 与 repository
- 再补 route 与 schema
- 避免先做前端再倒推接口

### 阶段三：前端接入

- 先补 `types/` 和 `lib/api.ts`
- 再补页面和组件
- 页面必须有加载态、空态、错误态

### 阶段四：文档与运行方式收口

- README
- `.env.example` 或 `.env.local.example`
- Docker / run.sh
- 手工验收说明

## 4. 任务拆分规则

推荐拆法：

- 一个任务只负责一个接口
- 一个任务只负责一个页面族
- 一个任务只负责一个 provider 层能力
- 一个任务只负责一个结构化整理能力

不推荐的拆法：

- 一个任务同时改 docs、后端、前端、README 且无明确主题
- 一边改数据模型，一边顺手重写半个页面
- 没更新契约，就先把前后端字段都加上

## 5. 并行协作规则

可以并行：

- provider 实现
- 前端页面实现
- 智能获取细节优化
- 测试和验收脚本
- 样例数据整理

需要串行：

- API 契约变更
- 数据模型变更
- 共享前端类型变更
- Docker / README 最终收口

并行时要求：

- 两个任务不要同时修改同一个共享契约文件
- 不要在 docs 没更新前新增共享字段
- 每个任务尽量收敛在有限文件集内

## 6. 交付标准

一项改动可以视为完成，至少要满足：

- 功能跑通
- 分层合理
- 文档同步
- 没有明显破坏现有能力
- 有基本验证

验证方式可以是：

- 自动测试
- 类型检查
- 冒烟脚本
- 明确的人工验收记录

## 7. 提交前自查

提交前至少检查：

- 是否误提交 `.env`、数据库、Cookie、下载内容
- 是否有本地缓存文件混进来
- 是否改了接口却没改文档
- 是否页面只有 happy path，没有错误态
- 是否 README 还停留在旧能力描述
